# Architecture — hermes-agent-ha-addon

> **Keep this document in sync with every structural change.**
> Future AI agents: read this before touching `run.sh`, `Dockerfile`,
> or `config.yaml`.  Also read `UPGRADE_LOG.md` for the *why* behind
> every non-obvious decision.

---

## 1. Component overview

```
HA Supervisor (Ingress proxy)
        │  single TCP port 8099
        ▼
┌─────────────────────────────────────────────┐
│  server.py  (Python http.server, port 8099) │  ← our ingress glue
│  Routes:                                    │
│   /          → index.html (launcher page)   │
│   /panel/**  → reverse proxy 127.0.0.1:9119 │
│   /ttyd/**   → reverse proxy 127.0.0.1:7681 │
│   /health    → 200 OK                       │
│   /config-model → reads HERMES_HOME/config.yaml│
└──────┬──────────────────────┬───────────────┘
       │                      │
       ▼                      ▼
hermes dashboard          ttyd (port 7681)
(port 9119)               bash -i in /config/workspace
upstream Hermes Web UI    writable terminal
       │
       ▼
hermes gateway run  (foreground, PID 1 via exec)
Nous Tool Gateway — manages sessions, HA platform,
skills, cron, messaging
```

## 2. Process startup order (run.sh)

1. Export env vars (`ADDON_STATE_ROOT`, `HERMES_HOME`, `HOME`, …)
2. `mkdir -p` persistent dirs
3. Python inline block — reads `/data/options.json`, writes
   `HERMES_HOME/.env` and `HERMES_HOME/config.yaml`
4. `source HERMES_HOME/.env` (set -a / set +a)
5. `unset MESSAGING_CWD` (belt-and-suspenders, see UPGRADE_LOG §v0.10.4)
6. Start **ttyd** on port 7681 in background
7. Start **server.py** in background
8. Start **hermes dashboard** on 127.0.0.1:9119 in background
9. Run **skills_sync.py** (bundles built-in skills into HERMES_HOME)
10. `exec hermes gateway run` — replaces shell; becomes the main process

> **CRITICAL:** Do NOT call `/opt/hermes/docker/entrypoint.sh`.
> It hardcodes `HERMES_HOME=/opt/data`, overwriting our export and
> wiping auth state.  See `UPGRADE_LOG.md §v0.9.9` and
> `memory/project_hermes_addon_hermes_home_landmine.md`.

## 3. Persistent storage layout (v0.11+)

Map type: **`addon_config:rw`** — container `/config` ↔ host
`/addon_configs/<slug>_hermes_agent/` (HA 2023.11+ standard, isolated
per add-on, survives uninstall).

```
/config/                          ← addon_config:rw mount root
├── .hermes/                      ← HERMES_HOME (Hermes state)
│   ├── .env                      ← generated by run.sh on every boot
│   ├── config.yaml               ← generated by run.sh on every boot
│   ├── SOUL.md                   ← agent personality (seeded from upstream)
│   ├── sessions/                 ← conversation history
│   ├── memories/                 ← agent long-term memory
│   ├── skills/                   ← bundled + user skills
│   ├── hooks/                    ← event hooks
│   ├── cron/                     ← scheduled tasks
│   ├── logs/                     ← gateway logs
│   └── .addon-runtime            ← TTYD_CWD sidecar (bash-only, not exported)
├── auth/                         ← AUTH_STORAGE_PATH
│   └── session.json              ← auth bridge state
└── workspace/                    ← default terminal cwd / messaging_cwd
```

`/data/options.json` — Supervisor-managed; read-only input from config UI.
`/data/` itself is ephemeral (cleared on Rebuild).

Host path: `ls /addon_configs/` on HA OS SSH to find the exact `<slug>`.

## 4. URL routing detail

All traffic enters on port **8099** through the HA Ingress proxy.  The
Ingress strips the `/api/hassio_ingress/<token>` prefix and forwards
bare paths to `server.py`.

| Path pattern | Handler |
|---|---|
| `/` | `index.html` (launcher with Dashboard + Terminal buttons) |
| `/styles.css`, `/app.js` | static files |
| `/health` | 200 OK (Ingress health check) |
| `/config-model` | reads `HERMES_HOME/config.yaml → model.default` |
| `/panel/**` | reverse proxy → `127.0.0.1:9119` (hermes dashboard) |
| `/ttyd/` | reverse proxy → `127.0.0.1:7681` (ttyd HTTP) |
| `/ttyd/ws` | WebSocket tunnel → ttyd (server.py rewrites origin headers) |
| `/shim/**` | OpenAI shim (provider_shim.py, web-login mode) |

## 7. Home Assistant 2026.x build compatibility

- Supervisor 2026.04.0+ no longer provides `BUILD_FROM` by default.  Keep the
  pinned upstream Hermes image in `Dockerfile` (`ARG BUILD_FROM=...`) and do
  not rely on a missing `build.yaml`.
- `build.yaml` is intentionally absent; Dockerfile labels must carry the HA
  app metadata (`io.hass.*`) required by the modern BuildKit path.
- `config.yaml` exposes `/health` through `watchdog` so Supervisor can detect
  an ingress wrapper that failed to start.

## 8. Common pitfalls checklist (read before every upgrade)

- [ ] Does the new upstream image still have `hermes dashboard`?
      (`hermes dashboard --help` in a test container)
- [ ] Does `hermes gateway run` still work without calling entrypoint.sh?
- [ ] Does the HA WebSocket URL still need the `/api/websocket` patch?
      (grep `api/websocket` in upstream homeassistant.py)
- [ ] Do the Dockerfile ENV paths match run.sh exports?
- [ ] Does `HERMES_HOME` inside the running gateway process point to
      `/config/.hermes`?
      `docker exec <addon> env | grep HERMES_HOME`
- [ ] Does `/config/.hermes/config.yaml` still have `model:` as a dict?
- [ ] Does `curl http://127.0.0.1:8099/config-model` return the real model?
- [ ] Are config.yaml `options:` and `schema:` in sync with
      `translations/en.yaml` and `translations/zh.yaml`?
